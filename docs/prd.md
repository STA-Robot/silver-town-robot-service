# 실버타운 물품 배분/회수 서비스 로봇 PRD

## 문서 목적

이 문서는 실버타운에서 서비스 로봇이 물품 배분, 물품 회수, 회수 물품 분류/정리를 수행하는 제품 시나리오의 Single Source of Truth이다.

기능 구현, Open-RMF task 설계, 로봇 주행/로봇팔 연동, UI/API 설계 문서는 이 PRD의 업무 흐름과 정책을 기준으로 작성한다. 구현 세부는 별도 문서에서 다루되, 사용자 시나리오와 제품 정책이 충돌할 경우 이 문서를 우선한다.

## 제품 개요

실버타운 입주자에게 물품을 나눠주고, 사용 후 물품을 회수하고, 회수된 물품을 창고에서 종류별로 분류/정리하는 로봇 서비스이다.

서비스는 다음 장비와 소프트웨어가 협력해서 동작한다.

- 물품 적재가 가능한 Pinky Pro 주행로봇
- 손 제스처를 인식하는 vision 모듈
- 테이블별 회수 호출 입력 장치
- 회수 물품을 하역/분류하는 고정식 JetCobot 로봇팔 최대 2대
- 로봇 task, traffic, 복귀 동작을 조정하는 Open-RMF 기반 관제/오케스트레이션 계층

## 하드웨어 구성

v1 서비스는 다음 하드웨어 구성을 기준으로 설계한다.

- 주행로봇: Pinky Pro 2대
- 로봇팔: JetCobot 2대
- 제어 보드: Raspberry Pi 4

Pinky Pro는 물품 배분/회수 주행, 대기 장소 복귀, 창고 이동을 담당한다. JetCobot은 창고 하역장소에서 회수 물품 하역/분류를 담당한다. Raspberry Pi 4는 로봇별 제어 보드로 사용하며, 로봇 adapter, 센서/액추에이터 연동, 현장 네트워크 통신을 수행한다.

## 목표

- 요양사가 물품을 배분할 때 무거운 적재물을 직접 운반하지 않도록 한다.
- 입주자가 테이블에서 간단한 호출만으로 물품 회수를 요청할 수 있게 한다.
- 회수된 물품이 가득 찬 로봇은 창고로 이동해 로봇팔 하역/분류를 거친 뒤 다시 서비스 대기 상태로 돌아오게 한다.
- 주행로봇, 요양사, 입주자, 로봇팔의 동작을 하나의 mission 흐름으로 추적할 수 있게 한다.

## 비목표

- 입주자 개인 식별, 건강 상태 판단, 복약 판단은 본 서비스 범위에 포함하지 않는다.
- 로봇이 입주자에게 직접 물품을 건네주는 동작은 v1 범위에 포함하지 않는다. 물품 배분은 요양사가 수행한다.
- 테이블 위 물품을 로봇이 직접 집어오는 동작은 v1 범위에 포함하지 않는다. 회수 물품 적재는 입주자 또는 현장 인력이 수행한다.
- 창고 외 구역에서 로봇팔이 하역/분류하는 동작은 v1 범위에 포함하지 않는다.

## 주요 사용자와 시스템

### 요양사

- 물품 배분 시나리오에서 주행로봇을 동행시킨다.
- 로봇에 적재된 물품을 입주자에게 나눠준다.
- 손 제스처로 주행로봇에 정지, 주행 재개, 종료 명령을 내린다.

### 실버타운 입주자

- 테이블에 앉아 서비스를 받는다.
- 물품 회수가 필요할 때 테이블 호출 장치를 누른다.
- 회수 로봇이 도착하면 물품을 로봇 보관함에 넣는다.

### 주행로봇

- 물품을 적재하고 사람을 따라 이동한다.
- 회수 call이 발생한 테이블로 자율주행한다.
- 정해진 시간 동안 정지 대기한다.
- 보관함 상태에 따라 대기 장소 또는 창고로 이동한다.

### 고정식 로봇팔

- 창고 하역장소에 도착한 주행로봇의 보관함에서 물품을 꺼낸다.
- 물품 종류에 따라 각자의 하역장소에 분류해 정리한다.
- 동시에 운용 가능한 로봇팔 수는 최대 2대이다.

### 오케스트레이터/RMF

- table call, 배분 종료, 보관함 상태, 창고 하역 완료 같은 이벤트를 mission 단위로 관리한다.
- 주행로봇 task를 생성하고, 로봇 배정과 복귀 흐름을 제어한다.
- 창고 하역이 필요한 경우 로봇팔 작업 완료를 기다린 뒤 mission을 완료한다.

## 핵심 용어

- `mission`: 사용자가 기대하는 하나의 업무 완료 단위. 예: 테이블 3번 회수 요청 처리.
- `task`: RMF 또는 로봇팔 adapter에 제출되는 실행 단위. 예: 테이블로 이동 후 30초 대기.
- `table call`: 테이블에서 발생한 회수 요청.
- `routing batch`: 여러 테이블 회수 call을 하나의 라우팅 단위로 묶은 작업 집합.
- `time window`: 첫 회수 call이 들어온 시점부터 특정 시간 동안 추가로 들어온 call을 같은 routing batch에 포함하기 위한 시간 구간.
- `following`: 주행로봇이 요양사를 따라 이동하는 모드.
- `대기 장소`: 회수 call이 없을 때 주행로봇이 머무는 기본 위치.
- `복귀`: mission 완료 후 주행로봇이 대기 장소로 돌아가는 동작.
- `창고`: 보관함이 가득 찬 로봇이 하역/분류를 위해 이동하는 장소.
- `하역장소`: 창고 내에서 주행로봇이 정지하고 로봇팔 작업을 받는 위치.
- `보관함 full`: 주행로봇이 추가 회수 call을 수행하기 전에 창고 하역이 필요한 상태.

## 공통 정책

- 기본 정지/대기 시간은 30초이다.
- 30초는 운영 설정값으로 조정할 수 있지만, 별도 요구가 없으면 제품 기본값은 30초이다.
- 주행로봇은 사람, 입주자, 테이블, 다른 로봇과 충돌하지 않도록 주행해야 한다.
- mission 상태는 추적 가능해야 하며, 실패/중단/수동개입 필요 상태를 구분해야 한다.
- 동일 mission에서 테이블 회수 후 창고 이동이 필요하면, 테이블 회수를 수행한 같은 주행로봇이 창고로 이동해야 한다.
- mission 완료 후 주행로봇은 대기 장소로 복귀해야 한다.
- 여러 테이블 회수 call이 동시에 또는 짧은 시간 안에 발생하면, 시스템은 발생한 모든 call의 테이블을 방문해야 한다.
- v1에서는 여러 call을 개별 mission/task로 순차 처리할 수 있다. 단, call이 누락되거나 후속 call이 기존 call을 덮어쓰면 안 된다.
- 향후 개선에서는 여러 목적지를 하나의 routing batch로 묶고, TSP/VRP와 유사한 최적화 방식으로 방문 순서와 로봇 배정을 결정한다.
- routing batch의 기본 후보는 첫 call 발생 시점부터 특정 time window 동안 들어온 call 집합이다.

## 시나리오 1: 물품 배분

### 목적

요양사가 물품이 적재된 주행로봇과 함께 이동하며, 테이블에 앉아 있는 입주자들에게 물품을 나눠준다.

### 기본 흐름

1. 요양사가 물품 배분 업무를 시작한다.
2. 물품이 적재된 주행로봇이 following 모드로 요양사를 따라간다.
3. 요양사는 테이블에 앉아 있는 입주자에게 로봇에 적재된 물품을 나눠준다.
4. 요양사가 손 제스처로 정지 명령을 내리면 로봇은 즉시 정지한다.
5. 정지 상태에서 30초가 지나면 로봇은 following 주행을 다시 시작한다.
6. 30초가 지나기 전이라도 요양사가 손 제스처로 주행 명령을 내리면 로봇은 following 주행을 다시 시작한다.
7. 요양사가 손 제스처로 종료 명령을 내리면 배분 mission을 종료하고 로봇은 복귀한다.

### 기능 요구사항

- 로봇은 요양사를 따라가는 following 모드를 지원해야 한다.
- vision 모듈은 최소한 `정지`, `주행`, `종료` 손 제스처를 분류해야 한다.
- 정지 명령을 받은 로봇은 주행을 멈추고 정지 타이머를 시작해야 한다.
- 정지 상태에서 30초가 지나면 별도 명령이 없어도 주행을 재개해야 한다.
- 정지 상태에서 주행 명령을 받으면 30초 타이머가 끝나지 않았더라도 즉시 주행을 재개해야 한다.
- 종료 명령을 받은 로봇은 following을 중단하고 대기 장소로 복귀해야 한다.
- 종료 명령은 자동 주행 재개보다 우선순위가 높아야 한다.

### 상태

- `distribution_ready`: 물품 적재 완료, 배분 시작 가능
- `following`: 요양사를 따라 이동 중
- `paused_by_gesture`: 정지 제스처로 정지 중
- `returning`: 종료 명령 후 대기 장소로 복귀 중
- `completed`: 복귀 완료
- `intervention_required`: 제스처 인식 실패, 주행 불가, 안전 문제 등으로 수동개입 필요

### 예외/엣지 케이스

- 정지 상태에서 종료 명령이 들어오면 즉시 복귀 흐름으로 전환한다.
- following 중 요양사를 추적하지 못하면 안전 정지 후 수동개입 필요 상태로 전환한다.
- 제스처 인식 신뢰도가 기준 이하이면 명령을 실행하지 않고 현재 안전 상태를 유지한다.
- 복귀 중 새 회수 call을 처리할 수 있는지는 운영 정책에 따른다. 회수 시나리오에서는 대기 또는 복귀 중인 로봇이 call 후보가 될 수 있다.

## 시나리오 2: 물품 회수

### 목적

입주자가 테이블에서 회수 요청을 생성하면, 주행로봇이 해당 테이블로 이동해 물품 적재를 기다리고, 보관함 상태에 따라 대기 장소 또는 창고로 이동한다.

### 기본 흐름

1. 입주자 또는 현장 인력이 테이블의 버튼 같은 호출 장치를 누른다.
2. 시스템은 해당 테이블에 대한 회수 call을 생성한다.
3. 여러 테이블에서 call이 동시에 발생하면 시스템은 모든 call을 처리 대상으로 등록한다.
4. 대기 중이거나 복귀 중이던 주행로봇 중 적합한 로봇이 call을 배정받는다.
5. 배정된 주행로봇은 해당 테이블로 자율주행한다.
6. 로봇이 테이블에 도착하면 정지 상태로 30초 동안 대기한다.
7. 입주자는 대기 시간 동안 회수할 물품을 로봇 보관함에 담는다.
8. 30초 대기 후 시스템은 보관함 상태를 확인한다.
9. 보관함에 여유가 있고 아직 방문하지 않은 배정 테이블이 있으면 다음 테이블로 이동한다.
10. 보관함에 여유가 있고 더 방문할 테이블이 없으면 로봇은 대기 장소로 복귀한다.
11. 보관함이 꽉 찼으면 로봇은 창고로 이동한다.
12. 창고에서 하역/분류를 마친 뒤 로봇은 대기 장소로 복귀한다.

### 기능 요구사항

- 테이블 호출 장치는 테이블 식별자와 함께 회수 call을 생성해야 한다.
- 회수 call은 mission으로 기록되어야 한다.
- 여러 테이블 회수 call이 동시에 들어오면 시스템은 각 call을 독립적으로 추적하고, 모든 대상 테이블을 방문해야 한다.
- 시스템은 대기 중인 로봇뿐 아니라 복귀 중인 로봇도 회수 call 배정 후보로 고려할 수 있어야 한다.
- 로봇은 배정된 테이블 waypoint까지 자율주행해야 한다.
- 하나의 로봇에 여러 테이블이 배정된 경우, 로봇은 보관함이 full이 되기 전까지 배정된 테이블을 순차 방문해야 한다.
- 향후 라우팅 최적화에서는 첫 call 이후 지정된 time window 안에 들어온 call들을 하나의 routing batch로 묶어, 방문 순서와 로봇 배정을 최적화해야 한다.
- 로봇은 테이블 도착 후 기본 30초 동안 정지 대기해야 한다.
- 대기 시간 동안 입주자가 물품을 넣을 수 있도록 로봇은 움직이지 않아야 한다.
- 대기 완료 후 보관함이 full인지 판단해야 한다.
- 보관함에 여유가 있고 더 방문할 테이블이 없으면 mission을 완료하고 로봇을 대기 장소로 복귀시켜야 한다.
- 보관함이 full이면 같은 로봇을 창고로 보내야 한다.
- 여러 테이블이 배정된 상태에서 보관함이 full이 되면, 아직 방문하지 않은 call은 완료 처리하지 않고 pending 상태로 유지하거나 다른 가용 로봇에 재배정해야 한다.
- 창고 하역/분류 완료 전까지 full 상태의 로봇은 신규 테이블 회수 call을 수행하면 안 된다.

### 상태

- `table_call_received`: 테이블 회수 call 생성
- `table_task_assigned`: 회수 로봇 배정 완료
- `moving_to_table`: 테이블로 이동 중
- `waiting_at_table`: 테이블 도착 후 정지 대기 중
- `checking_storage`: 보관함 상태 확인 중
- `returning`: 대기 장소로 복귀 중
- `moving_to_warehouse`: 보관함 full로 창고 이동 중
- `waiting_for_unload`: 창고 하역/분류 대기 또는 진행 중
- `completed`: mission 완료
- `intervention_required`: 수동개입 필요

### 예외/엣지 케이스

- 동일 테이블에서 중복 call이 발생하면 진행 중인 mission이 있는지 확인하고 중복 mission 생성을 방지해야 한다.
- 서로 다른 여러 테이블에서 call이 동시에 발생하면 중복으로 간주하지 않고 모두 방문 대상으로 유지해야 한다.
- 로봇이 테이블로 이동할 수 없으면 다른 로봇으로 재배정하거나 수동개입 필요 상태로 전환해야 한다.
- 테이블 도착 후 물품 적재가 30초 안에 끝나지 않는 운영 상황은 v1에서는 자동 연장하지 않는다. 단, 향후 UI/API 또는 추가 입력 장치로 대기 연장을 지원할 수 있다.
- 보관함 상태를 확인할 수 없으면 안전하게 full 또는 수동개입 필요로 처리하는 정책을 선택해야 한다.
- 창고 이동 중 장애가 발생하면 mission은 완료되지 않은 상태로 유지되어야 한다.

## 시나리오 3: 회수된 물품 분류/정리

### 목적

보관함이 꽉 찬 주행로봇이 창고에 도착하면, 고정식 로봇팔이 회수 물품을 종류별로 정해진 하역장소에 분류해 정리한다.

### 기본 흐름

1. 보관함 full 상태의 주행로봇이 창고로 이동한다.
2. 주행로봇은 창고의 하역장소에 정지한다.
3. 시스템은 사용 가능한 로봇팔을 최대 2대까지 배정한다.
4. 로봇팔은 주행로봇 보관함에서 물품을 꺼낸다.
5. 로봇팔은 물품 종류를 기준으로 각자의 하역장소에 분류해 하역한다.
6. 모든 대상 물품의 하역/분류가 완료되면 시스템은 주행로봇의 보관함을 비움 상태로 갱신한다.
7. 주행로봇은 대기 장소로 복귀한다.

### 기능 요구사항

- 창고 하역장소는 주행로봇이 로봇팔 작업 가능 위치에 정지할 수 있도록 정의되어야 한다.
- 로봇팔은 동시에 최대 2대까지 사용 가능해야 한다.
- 로봇팔은 물품 종류별 목적 하역장소를 구분해야 한다.
- 로봇팔 작업 완료 여부는 mission 상태에 반영되어야 한다.
- 하역이 성공적으로 끝나야 주행로봇의 보관함 상태를 full이 아닌 상태로 전환할 수 있다.
- 하역 완료 후 주행로봇은 대기 장소로 복귀해야 한다.

### 상태

- `warehouse_arrived`: 주행로봇이 창고 하역장소에 도착
- `arm_assignment_pending`: 사용 가능한 로봇팔 배정 대기
- `unloading`: 로봇팔 하역/분류 진행 중
- `unload_completed`: 하역/분류 완료
- `returning`: 주행로봇 복귀 중
- `completed`: mission 완료
- `intervention_required`: 로봇팔 실패, 물품 인식 실패, 하역 실패 등으로 수동개입 필요

### 예외/엣지 케이스

- 사용 가능한 로봇팔이 없으면 주행로봇은 창고 하역장소 또는 지정 대기 위치에서 대기해야 한다.
- 로봇팔 1대만 사용 가능하면 1대로 하역을 진행할 수 있어야 한다.
- 물품 종류를 판단할 수 없는 경우 미분류 하역장소로 보내거나 수동개입 필요 상태로 전환하는 정책이 필요하다.
- 로봇팔 하역이 실패하면 주행로봇은 복귀하지 않고 수동개입 필요 상태로 남아야 한다.

## 우선순위

### P0

- 물품 배분 following 시작/정지/주행/종료 제스처 처리
- 정지 후 30초 자동 주행 재개
- 종료 후 로봇 복귀
- 테이블 회수 call 생성
- 동시 또는 근접 시간대에 발생한 여러 테이블 call의 누락 없는 등록과 방문
- 회수 로봇 테이블 이동
- 테이블 도착 후 30초 정지 대기
- 보관함 full 여부에 따른 대기 장소 복귀 또는 창고 이동
- 창고 도착 후 로봇팔 하역 완료 신호 수신
- 하역 완료 후 로봇 복귀

### P1

- 복귀 중인 로봇의 회수 call 재배정 최적화
- 중복 table call 방지
- 로봇팔 2대 병렬 하역
- 물품 종류별 하역장소 매핑 관리
- mission 상태 UI/API 조회
- 실패 사유와 수동개입 필요 상태 표시

### P2

- 테이블 대기 시간 연장
- 제스처 인식 신뢰도/히스토리 대시보드
- 회수량/하역량 기반 운영 리포트
- 물품 종류별 재고 또는 처리량 통계
- time window 기반 routing batch 구성
- TSP/VRP 유사 최적화를 통한 다중 목적지 방문 순서 및 로봇 배정 개선

## 성공 기준

- 요양사가 손 제스처만으로 배분 로봇을 정지, 주행 재개, 종료시킬 수 있다.
- 배분 중 정지된 로봇은 기본 30초 후 자동으로 following을 재개한다.
- 테이블 call 발생 후 로봇이 해당 테이블로 이동해 30초 동안 안정적으로 대기한다.
- 여러 테이블 call이 동시에 발생해도 모든 대상 테이블이 방문된다.
- 보관함에 여유가 있는 회수 mission은 테이블 대기 후 대기 장소 복귀까지 완료된다.
- 보관함이 full인 회수 mission은 같은 로봇이 창고로 이동하고, 하역 완료 후 복귀한다.
- 창고 하역 시 최대 2대의 로봇팔이 물품을 종류별 하역장소로 분류할 수 있다.
- 각 mission은 현재 상태와 완료/실패 여부를 추적할 수 있다.

## 미결정 사항

- 테이블 호출 장치의 정확한 형태: 물리 버튼, 터치 UI, QR/API 등
- 보관함 full 판단 방식: 무게 센서, 칸별 센서, vision, 수동 입력 등
- 물품 종류 판별 방식: vision, 태그, 바코드, 수동 분류 등
- 배분 시나리오에서 복귀 중인 로봇이 회수 call을 받을 수 있는지에 대한 운영 우선순위
- 테이블 대기 30초 후 물품 적재가 끝나지 않았을 때의 연장 정책
- routing batch를 구성할 time window의 기본 길이
- 다중 테이블 call을 한 로봇이 순회할지, 여러 로봇에 분산할지 결정하는 배정 정책
- 로봇팔 하역 실패 시 재시도 횟수와 수동개입 알림 채널
- 창고 하역장소가 1개인지, 로봇팔별로 별도 하역장소를 둘지 여부

## 문서 변경 원칙

- 제품 시나리오, 기본 정책값, mission 흐름을 바꾸는 변경은 이 PRD를 먼저 갱신한다.
- 구현 문서는 이 PRD의 요구사항을 만족하는 구체 설계로 작성한다.
- 코드의 임시 구현값이 이 PRD와 다를 경우, 구현값을 제품 기본 정책으로 간주하지 않는다.
- 운영 실험으로 정책값을 바꿀 수는 있지만, 정식 제품 정책으로 확정하려면 이 문서를 갱신해야 한다.
